一、Kubernetes核心概念

1 什么是Kubernetes？

Kubernetes 最初源于谷歌内部的Borg，Borg是谷歌内部的大规模集群管理系统，负责对谷歌内部很多核心服务的调度和管理，Borg的目的是让用户能够不必操心资源管理的问题，让他们专注于自己的核心业务，并且做到跨多个数据中心的资源利用率最大化。实现自动装箱，自我修复，水平扩展，服务发现和负载均衡，自动发布和回滚，密钥和配置管理，存储编排。

Kubernetes就是一个集群，整合多台云平台主机的资源，对外提供对应的服务。在多台主机安装Kubernetes相关应用程序，通过这个应用进行协同通信工作，让我们更加应用简单、高效快捷的容器管理平台。

2 组件介绍

2.1 Master三大组件：

2.1.1 Kubernetes-ApiServer：

是k8s整个访问的入口，提供了k8s各类资源对象的增删改查即watch等HTTP RESTFUL接口。 1. 端口默认值为6443，可通过--secure-port的值进行修改默认值 2. 默认IP地址为非本地(Non-Localhost)网络端口，通过启动参数--bind-address设置该值。 3. 该端口用于接收客户端，dashboard等外部HTTPS请求。 4. 用于基于Token文件或客户端证书及HTTP Base的认证。 5. 用于基于策略的授权。 -> 800x500

2.1.2 kube-schduler

是一个控制面进程，负责将Pod指派到节点上。通过调度算法为待调度Pod列表的每个Pod从可用Node列表中选择一个最合适的Node，并将信息写入到etcd中。 node节点上的kubelet通过每隔1s就会和API Server通信机制，监听到kubernetes Schduler产生到etcd中的Pod绑定信息，然后获取对应的Pod清单，下载Image，并启动容器。 分配策略：1. 根据资源分配(CPU+内存)。 2. 根据label分配。3. 根据资源使用率最均衡的节点

2.1.3 kube-controller-manager

是一个管理面进程。包括一些子控制器（副本控制器、节点控制器、命名空间控制器和服务账号控制器等），控制器作为集群内部的管理控制中心，负责集群内的Node、Pod副本、服务端点、命名空间、服务账号、资源定额的管理。并且controller-manager控制器每隔5s检查一次节点状态，如果controller-manager没有收到节点的心跳，则等待40s后，就将该节点标记为不可达，如果该节点在标记后5分钟没有回复，controller-manager会删除当前node节点的所有pod并在其他可用节点重建这些pod。

2.1.4 创建容器流程

  1. kubelet客户端 发起请求3个Pod请求到 Api Server
  2. Api Server 收到指令，把信息写入到 etcd
  3. kube-scheduler 通过与Api Server 进行通信检测到有3个Pod未创建。
  4. kube-scheduler 将 etcd中的3个pod一次调度获取到，然后检测是否携带Table标签等，如果没有携带，就按照默认规则分配(检查 节点中的资源谁更符合)。
  5. 一旦在node节点搭建一个Pod服务，Api Server 就会通过心跳检测到node节点的资源更新，然后将node节点的更新数据更新到etcd

2.1.5 Pod高可用机制

  • node monitor period：节点监视周期
  • node monitor grace period：节点监视器宽限期
  • pod eviction timeout：pod驱逐超时时间

2.2 Etcd Cluster

用来存放k8s集群信息和用来实现kube-schduler、kube-controller-manager，workNode之间的通信的。并且通过key-value数据存储系统的，生产环境使用时需要为etcd提供备份功能。

2.3 Work Node 组件：

2.3.1 kubelet

kubelet通过Api Server监听到kubernetes Scheduler产生的Pod绑定信息，然后操控Docker获取对应的Pod清单，下载Image，并启动容器。是运行在每个worker节点的代理组件，他会监视已分配给节点的pod

  1. 向master汇报node节点的状态信息。
  2. 接受指令并再pod中创建docker容器
  3. 准备Pod所需的数据卷
  4. 返回pod的运行状态。
  5. 在node节点执行容器健康检查。

2.3.2 kube-proxy

用来部署到每个Node节点上的网络代理，它反映了node上Kubernetes API 中定义的服务，ing可以通过一组后端进行简单的TCP、UDP 和 SCTP流转发或者在一组后端进行循环转发，用户必须使用 ApiServer API 创建一个服务来配置代理，其实就是kube-proxy通过在主机上维护网络规则并执行连接转发来实现Kubernetes服务访问。 作用：监听API SERVER 中服务对象的变化， 再通过管理 IPtables或者IPVS规则来实现网络的转发。

Kube-Proxy 不同版本可支持的三种工作模式：

版本 支持模式
k8s v1.1 UserSpace
K8s v1.2 IPtables
K8s v1.9~v1.11 IPVS、IPtables
注意：IPVS 要安装ipvsadm、ipset工具包和加载 ip_vs 内核模块。并且当IPVS配置写错或者内核不支持，Kube-Proxy会自动降级为IPtables。

Kube-proxy 有三个网络：

  • node 网络
  • pod 网络
  • service 网络

2.3.3 Dasboard

是就网页的Kubernetes用户界面，可以使用Dashboard获取运行中

2.3.4 coreDNS

负责为整个集群提供DNS服务，从而实现服务之间的访问。

    state MASTER   #当前节点在此虚拟路由器上的初始状态，状态为MASTER或者BACKUP，一般都是配置backup，该值无法决定身份，最终还是通过比较priority
    interface enp4s0    #绑定为当前虚拟路由器使用的物理接口，如：ens32,eth0,bond0,br0
    garp_master_delay 10
    smtp_alert
    virtual_router_id 56    #每个虚拟路由器惟一标识，范围：0-255，同一组虚拟路由器的vrid必须一致
    priority 50              #当前物理节点在此虚拟路由器的优先级，范围：1-254，每个keepalived主机节点此值不同
    advert_int 1          #vrrp通告的时间间隔，默认1s
    authentication {       #认证机制
        auth_type PASS     #AH（不推荐）或PASS
        auth_pass 1111     #预共享密钥，仅前8位有效，同一个虚拟路由器的多个keepalived节点必须一样
    }
    virtual_ipaddress {    #虚拟IP
