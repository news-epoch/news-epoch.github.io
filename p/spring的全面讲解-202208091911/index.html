Spring的历史

[!timeline] Spring

Rod Jahnson首次推出了Spring框架雏形interface21框架。

2004年3月

Spring框架以interface21框架为基础，经过重新设计，发布了1.0正式版。

Spring是为了什么而创建出来的？

Spring是为了简化企业级开发而出现的框架。它可以将Spring对象、Dao组件对象、Service组件对象等交给Spring容器进行管理，有效的降低代码的耦合度，方便项目的后期维护、升级和扩展。

Spring提供了下面几个特性用来简化开发工程师的开发

  1. 非侵入式：如果使用侵入式的框架，一般需要继承或者实现框架的某一个类或接口，这样你把框架拿到以后就会运行不起来，甚至在代码层面会报错哦。那么如同spring这种非侵入式的框架，都是利用反射和动态调用来实例化的，代码中没有任何与spring交叉的类，哪怕把spring的jar全部去了，你的代码不受任何影响，加上其他的框架后可以继续运行。
  2. 控制反转(IOC)(主要)：就是将对象的创建交给Spring去进行创建。
  3. 依赖注入(DI)：将对象中的属性不用手动调用setxx方法进行赋值，而是可以通过配置文件进行自动赋值。
  4. 面向切面编程(AOP)(主要)：在项目后期中，用来方便模块功能的扩展，降低耦合度，或者进行日志的打印。
  5. 容器：Spring是一个容器，所以它管理的对象是存在生命周期这个概念的。
  6. 组件化：组件也是抽象的概念,可以理解为一些符合某种规范的类组合在一起就构成了组件。
  7. 一站式：Spring可以通过IOC和AOP进行整合各种企业的开源框架，例如Mybatis，kafka等

Spring配置文件讲解

单实例和多实例

一、 当scope取值为singleton时

  1. Bean的实例化个数：1个
  2. Beande的实例化时机：当配置文件加载(new ClassPathXmlApplicationContext)时，就会进行创建
  3. Bean的生命周期：
    • 对象创建： 当应用加载时，就会根据配置文件中的内容进行初始化实例
    • 对象运行：只要容器在，对象一直活着。
    • 对象销毁：当应用卸载，销毁容器时，对象就被销毁了。

二、 当scope取值为Prototype时

  1. Bean的实例化个数：N个
  2. Beande的实例化时机：当配置文件加载完成后，调用getBean()方法时，就会进行创建
  3. Bean的生命周期：
    • 对象创建：使用容器对象调用getBean()方法时，就会根据配置文件中的内容进行初始化实例
    • 对象运行：只要容器对象在使用，实例就一直活着。
    • 对象销毁：只要容器对象长时间未使用实例，就会被java的垃圾回收期回收。

Bean实例化的三种方式

  • 无参构造方法实例化
  • 工厂静态方法实例化
  • 工厂实例方法实例化

数据源(连接池)的作用：

  • 数据源是提高程序性能而出现的。

  • 使用完毕后将连接资源归还给数据源。

注解方式管理程序

注解 说明
@Component("") 使用在类上用于实例化Bean、
@Controller("") 使用在web层类上用于实例化Bean
@Service("") 使用在service层类上用于实例化Bean
@Repository("") 使用在dao层类上用于实例化Bean
@Autowired() 使用在字段上用于根据类型依赖注入
@Qualifier("") 结合@Autowired一起使用用于根据名称进行依赖注入
@Resource(name="") 就相当于@Autowired+@Qualifier,按照名称进行注入
@Value("") 注入普通属性
@Scope("") 标注Bean的作用范围,例如propotype或singleton
@PostConstruct() 使用在方法上标注该方法时Bean的初始化方法
@PreDestroy() 使用在方法上标注该方法是Bean的销毁方法

Spring新注解

因为无法完全通过原始注解进行替代xml配置文件，例如：

  1. 加载外部文件内容<context:property-placeholder location="classpath:mysql_jdbc.properties"/>
  2. 非自定义的bean<bean id="dataSource" class="com.alibaba.druid.pool.DruidDataSource">
  3. 配置组件扫描<context:property-placeholder location="classpath:mysql_jdbc.properties"/>
  4. 引入外部spring配置文件<import resource="applicationContext.xml"></import>
注解 说明
@Configuration 用于指定当前类是一个Spring配置类，当创建容器时会从该类上加载注解
@ComponentScan 用于指定Spring在初始化容器时要扫描的包
@Bean 使用在方法上，标注将该方法的返回值存储到Spring容器中
@PropertySource 用于加载.properties文件中的配置
@Import 用于导入其他配置类

Spring集成junit

在测试类中，每个测试方法都有一下两行代码

1
2

ApplicationContext app = new AnnotationConfigApplicationContext(SpringConfiguration.class);  
UserService us = app.getBean(UserService.class);

以上两行代码作用是获取容器，如果不屑，直接会提示空指针异常，所以又不能轻易删掉。

上述问题解决思路

  • 让springjunit负责创建容器，但是需要将配置文件的名称告诉它
  • 将需要进行测试的Bean直接在测试类中注入。

Spring继承Junit步骤：

  1. 导入Spring继承junit的坐标，注意junit的坐标不得低于4.12
  2. 使用@Runwith()注解替换原来的运行周期
  3. 使用@ContextConfiguration(“classpath:")指定配置文件或配置类
  4. 使用@Autowired注入需要测试的对象
  5. 创建测试方法进行测试

AOP相关概念

Spring 的 AOP实现底层就是对上面的动态代理代码进行封装，封装后我们只需要对需要关注的部分进行代码编写，并通过配置的方式完成指定目标的方法增强。

  • Target（目标对象）：代理目标。
  • Proxy（代理）：一个类的AOP之如增强后，就产生一个结果代理类。
  • Joinpoint（连接点）：所谓连接点是指那些可以被拦截到的所有点。在spring中，这些点值的是方法，因为spring只支持方法类型的连接点。
  • Pointcut（切入点）：所谓切入点是指我们要对那些Joinpoint进行拦截的定义。
  • Advice（通知/增强）：所谓通知是指拦截到Joinpoint之后索要做的事情就是通知。
  • Aspect（切面）：是切入点和通知（引介）的结合。
  • Weaving（织入）：是指把增强应用到目标对象来创建新的代理对象的过程。spring采用动态代理织入，而AspectJ采用编译器织入和类装载期织入。

public class ProxyTest {  
    public static void main(String[] args) {  
        TargetInterface target = new Target();  //创建目标对象
        Advice advice = new Advice();  //创建通知
        TargetInterface proxy = (TargetInterface) Proxy.newProxyInstance(   //创建代理对象
                target.getClass().getClassLoader(),  //添加类加载器来定义代理类
                target.getClass().getInterfaces(),  //添加代理类实现的接口列表
                new InvocationHandler() {  //调度方法调用的调用处理函数
                    @Override  
                    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {  
                        advice.before();  
                        method.invoke(target, args);  
                        advice.after();  
                        return null;                    }  
                }  
        );  
        proxy.show();  
    }  
}

AOP开发

AOP开发的四种通知模式：

名称 代码 用法
前置通知 @Before() 用于配置前置通知，指定增强方法再切入点之前执行
后置通知 @AfterReturning() 用于配置后置通知，指定增强方法再切入点之后执行
环绕通知 @Around() 用于配置环绕通知，指定增强方法再切入点之前和之后执行
异常抛出通知 @AfterThrowing() 用于配置前置通知，指定增强方法再切入点出现异常执行
最终通知 @After() 用于配置前置通知，指定增强方法不论切入点是否出现异常都会执行
